\section{Caching in Link Traversal-based Query Processing}
To make this work self-contained, we first briefly outline the mechanics of Link Traversal-based Query Processing (LTQP).
Unlike traditional query engines that evaluate queries against a centralized,
pre-built index, LTQP evaluates queries directly over the live Web using the ``follow-your-nose'' principle of Linked Data. 
Execution relies on two core components: a \emph{link queue} and an \emph{active local store}.
The engine initializes the link queue with seed URIs (often extracted from the query).
It then iteratively dereferences these URIs, retrieving RDF documents.
The triples from these documents are deposited into the active local store; a temporary, query-specific,
in-memory dataset over which the SPARQL operators are actually evaluated. 
Simultaneously, new URIs are discovered within the retrieved data 
and added to the link queue according to specific \emph{reachability semantics},
which dictate which links are relevant to follow based on the query structure.
Furthermore, to optimize memory usage, 
engines like Comunica~\cite{taelman2018comunica} typically employ \emph{dictionary encoding} within the local store,
converting lengthy string-based URIs and literals into compact integer identifiers.

Hartig~\cite{hartig2011caching} originally established the fundamental concept of caching within this paradigm,
demonstrating that keeping a local repository of recently accessed Web documents mitigates
the high network latency inherent to LTQP.



To validate the query sequences generated by the SolidBenchSequence (SBS) benchmark,
we implemented three caching approaches in the Comunica link traversal engine~\cite{taelman2018comunica,taelman2023link}.
Specifically, we cache the triples from documents dereferenced during link traversal in-memory.
High cache hit rates eliminate redundant HTTP requests, increasing traversal speed and reducing query latency.

\paragraph{Indexed Cache}

\paragraph{Indexed Cache + Cardinality estimation}
Because the cache is indexed, it can execute count queries to estimate quad pattern cardinalities.
Traditional LTQP relies on zero-knowledge heuristics due to a lack of prior knowledge~\cite{hartig2011zero}.
By estimating cardinalities against \emph{all} cached documents,
regardless of their reachability for the current query,
we can replace these heuristics with Comunica's default cost-based optimizer~\cite{taelman2018comunica}
to generate more accurate query plans. \\

Our implementations adhere to the RFC-9111 specification~\cite{fielding2022rfc}, 
meaning the cache correctly interprets standard HTTP headers (like Cache-Control)
to determine how long to store data, just like a web browser.
However, because the benchmark dataset is static, cache entries require no revalidation and never become stale. 
Consequently, the observed hit rates are likely higher than in a live environment.
